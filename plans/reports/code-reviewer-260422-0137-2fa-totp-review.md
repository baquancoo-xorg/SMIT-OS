# Code Review: 2FA TOTP Implementation

**Date:** 2026-04-22
**Scope:** 10 files — TOTP service, auth service, schemas, routes, AuthContext, LoginPage, TwoFactorAuthTab, settings tabs/page, Prisma schema
**Stack:** Express 5 + Prisma + JWT httpOnly cookie + React 19 + TypeScript

---

## Overall Assessment

Solid implementation with correct architecture. Core security properties (AES-256-GCM encryption, bcrypt-hashed backup codes, scoped temp token, secret stripping) are all in place. Several issues below need fixing before this ships to production.

---

## Critical Issues

### 1. TOTP Secret trusted from client on enable — code verification uses unencrypted client input

**File:** `server/routes/auth.routes.ts` lines 150–153

```ts
// POST /2fa/enable
const { secret, code } = req.body;
if (!totpService.verifyCode(secret, code))   // secret comes from req.body!
  return res.status(400).json(...)
const encryptedSecret = totpService.encryptSecret(secret);
```

The `secret` field in the enable request body is the **plain base32 secret** originally generated by `GET /2fa/setup`. The client is asked to re-submit it, but the server has no way to verify this is the same secret it generated — an attacker who intercepts or guesses a valid secret + matching code can enroll any TOTP secret of their choosing.

**Fix:** Generate the secret server-side in `GET /2fa/setup`, store it temporarily (signed JWT or server-side session keyed to userId + short TTL), and on `POST /2fa/enable` retrieve it by userId — never trust the client to submit the secret.

Minimum safe alternative: drop the `secret` field from the enable schema, re-generate from the same seed stored in a short-lived server cache or an encrypted cookie tied to the session.

### 2. `requireAuth` middleware does NOT reject `totp-pending` tokens on 2FA management endpoints

**File:** `server/routes/auth.routes.ts` lines 20–27

`requireAuth` calls `authService.verifyToken`, which accepts any valid JWT including those with `purpose: 'totp-pending'`. Routes `/2fa/setup`, `/2fa/enable`, and `/2fa/disable` all use `requireAuth` but do not check `purpose`.

A user who completes only step 1 of login (password OK, TOTP not yet verified) holds a `totp-pending` token. They could POST to `/2fa/disable` to strip their own 2FA and then log in without the second factor.

**Fix:** In `requireAuth` (or a separate `requireFullSession` variant), reject tokens where `payload.purpose === 'totp-pending'`:

```ts
if (payload.purpose === 'totp-pending') return res.status(401).json({ error: 'Complete 2FA login first' });
```

## High Priority

### 3. No rate limiting on `/login/totp` — TOTP brute-forceable via repeated temp token reuse

The temp token is valid for 5 minutes. With 6-digit TOTP there are at most ~33 current valid codes across a ±1 window. An attacker who obtains a temp token can spray all possible 6-digit codes within 5 minutes with no throttle. The JWT has no per-user nonce, so it cannot be invalidated after one attempt.

**Fix (either):**
- Add express-rate-limit on `/login/totp` (e.g., 5 attempts per IP per minute).
- Store a per-user attempt counter in Redis/DB and lock after N failures.

### 4. `totpEnabled` state in `TwoFactorAuthTab` is local — desynchronises from actual server state

**File:** `src/components/settings/two-factor-auth-tab.tsx` line 19

```ts
const [totpEnabled, setTotpEnabled] = useState(currentUser?.totpEnabled ?? false);
```

`currentUser` is read once at mount. If the user opens two tabs and enables/disables 2FA in one, the other tab shows stale UI. More critically, on disable success the local flag is set to `false` but `currentUser` in `AuthContext` is never updated — so a page refresh (or any component that reads `currentUser.totpEnabled`) will show stale data.

**Fix:** After successful enable/disable, call `setCurrentUser` (from `useAuth`) or add a `refreshSession` helper that re-hits `/api/auth/me` and updates context.

### 5. Backup codes returned via JSON in login response leak count of remaining codes

**File:** `server/routes/auth.routes.ts` line 85

```ts
return res.json({ ...safeUser, backupCodeUsed: true, remainingBackupCodes: remaining.length });
```

`remainingBackupCodes` is an operational aid but it also tells an attacker exactly how many codes remain. If an attacker is burning backup codes they know when to stop trying. Minor data-discipline issue.

**Recommendation:** Remove `remainingBackupCodes` from the response, or show only a frontend-friendly boolean `lowBackupCodes: remaining.length <= 2`.

## Medium Priority

### 6. `crypto.ts` silently falls back to empty string key when both env vars are unset in non-production

**File:** `server/lib/crypto.ts` lines 4–11

```ts
const SECRET = process.env.APP_SECRET ?? process.env.JWT_SECRET ?? '';
if (SECRET.length < 32) {
  if (process.env.NODE_ENV === 'production') throw ...
  console.warn(...)
}
const key = crypto.createHash('sha256').update(SECRET).digest();
```

With an empty-string SECRET the server still starts, derives a deterministic key, and stores encrypted TOTP secrets in the database. In staging/CI environments that share a database with different secrets, previously encrypted rows become undecryptable with no warning. The `decrypt` call will throw at runtime, surfacing as a 500 to the user with no actionable error message.

**Fix:** Wrap `decrypt` calls in a try/catch in the routes and return `res.status(500).json({ error: 'Configuration error' })` — do not leak the raw exception. Also consider throwing in dev when SECRET is empty rather than just warning.

### 7. `signTempToken` includes `role: ''` and `isAdmin: false` — payload shape confusingly matches `JWTPayload`

**File:** `server/services/auth.service.ts` lines 35–41

The intent (confirmed by requirement) is that temp tokens should NOT contain role/isAdmin. They do include these fields with zeroed values, which means `verifyToken` (not `verifyTempToken`) would return a seemingly valid payload with empty role. Issue 2 above is the exploitable consequence. Separately, `role: ''` and `isAdmin: false` are semantically misleading.

**Fix:** Use a narrower interface `TempTokenPayload { userId: string; purpose: 'totp-pending' }` and avoid including role/isAdmin at all in the JWT claims.

### 8. `totpVerifySchema` max length 8 permits lowercase backup codes but `verifyAndConsumeBackupCode` normalises to uppercase

**File:** `server/schemas/auth.schema.ts` line 12 vs `server/services/totp.service.ts` line 56

Schema: `z.string().min(6).max(8)` — no character class restriction.
Service: `inputCode.replace(/s/g, '').toUpperCase()`

This is fine at runtime, but the schema allows arbitrary characters (letters, digits, symbols) between 6–8 chars. A backup code is exactly 8 hex characters uppercase (`crypto.randomBytes(4).toString('hex').toUpperCase()`). The schema should enforce this:

```ts
// backup codes are 8 hex chars, TOTP is 6 digits
code: z.string().regex(/^[0-9A-Fa-f]{8}$|^d{6}$/)
```

## Low Priority / Informational

### 9. Settings page (Security tab) is gated behind `isAdmin` — non-admin users cannot manage their own 2FA

**File:** `src/pages/Settings.tsx` lines 18–27

`if (!isAdmin) return <AccessDenied>`. `TwoFactorAuthTab` manages the **current user's** 2FA, which should be available to any authenticated user. If 2FA is a security requirement for all users, the tab needs to be accessible outside the admin settings wall.

This is a product decision, not necessarily a bug. But if non-admin users are expected to enroll 2FA, the current UX blocks them entirely.

### 10. `GET /2fa/setup` returns plain `secret` in the response body

By design, yes — the client needs it to display the setup flow. But the secret is only temporarily valid (before confirmation). No session binding means the secret floats in the browser indefinitely if the user abandons setup. Low risk for internal tooling but worth noting.

## Recommended Actions (Priority Order)

1. **[Critical]** Fix enable flow: never trust client-submitted secret. Cache generated secret server-side tied to session.
2. **[Critical]** Add `purpose !== 'totp-pending'` check in `requireAuth` or a dedicated `requireFullSession` guard applied to all `/2fa/*` routes.
3. **[High]** Add rate limiting on `POST /login/totp`.
4. **[High]** After 2FA enable/disable, refresh `currentUser` in AuthContext.
5. **[Medium]** Wrap `decrypt` in try/catch in routes; return clean 500, not raw exception.
6. **[Medium]** Use a narrower type for `signTempToken` payload — exclude role/isAdmin claims entirely.
7. **[Low]** Tighten `totpVerifySchema` code pattern to reject obviously invalid inputs.
8. **[Product]** Decide whether non-admin users need access to 2FA settings and move tab accordingly.

---

## Positive Observations

- AES-256-GCM with random IV per encrypt call is correct — no IV reuse risk.
- `verifyAndConsumeBackupCode` is constant-time in the sense that it iterates all codes regardless of early match (breaks early on first match, but this is acceptable for 8 codes).
- `safeUser` pattern consistently strips `password`, `totpSecret`, and `totpBackupCodes` across all login response paths.
- `requireAdmin` correctly checks `req.user.isAdmin` (populated from verified JWT), not a query parameter.
- Backup codes are 32 bits of entropy (hex of 4 random bytes) — acceptable for 8 one-time codes with bcrypt storage.
- `/me` endpoint correctly uses `select` to fetch only safe fields — no accidental leakage.
